Historical Case Study: The Ariane 5 Disaster

Date: June 4, 1996
The maiden flight of Europe's Ariane 5 rocket lasted only 40 seconds before veering off course and self-destructing.
The Culprit: A single software error - an integer overflow when converting a 64-bit floating-point number to a 16-bit signed integer.
The software was reused from Ariane 4 without proper testing for the new rocket's higher velocities.
üí∏ Total Loss: $370 Million
Source: ESA - "Ariane 5 Flight 501 Failure Report"

The Ariane 5 Lesson üìö

What Went Wrong?

  • TypeError The value exceeded what a 16-bit integer could hold
  • No Error Handling The system had no try-catch or validation
  • Assumption Failure Code worked on Ariane 4, assumed it would work on Ariane 5
  • Testing Gap The specific conditions weren't tested before launch

ü§î Reflection Question:

Key Takeaway: Small errors in understanding data types and limits can have catastrophic consequences. Always validate, handle errors, and test thoroughly! üéØ

Admiral Grace Hopper published "The Education of a Computer" - the first comprehensive paper on debugging.

  • ‚ú® Subroutines for testing - isolating code sections
  • ‚ú® Systematic debugging methods - step-by-step problem solving

Logical Errors ü§Ø

The sneakiest bugs

Logical Errors Are Harder

  • No error message appears!
  • Code runs successfully
  • But the result is wrong
  • Requires careful testing
// Calculate average - but there's a bug!
function average(numbers) {
  let sum = 0;
  for (let i = 0; i <= numbers.length; i++) {
    sum += numbers[i];
  }
  return sum / numbers.length;
}

console.log(average([10, 20, 30]));
// Returns NaN - but why? ü§î

Answer: i <= numbers.length should be i < numbers.length

Historical Tragedy: The Therac-25 Radiation Disaster

Timeline: 1985-1987
At least six patients received massive radiation overdoses (up to 100 times the intended dose) from the Therac-25 radiation therapy machine.
The Cause: Race conditions in the software that occurred only under specific timing circumstances.
The errors were nearly impossible to reproduce and debug because they depended on the exact timing of operator keystrokes.
Critical Failures:
  • ‚ùå No hardware safety interlocks (relied solely on software)
  • ‚ùå Inadequate error handling - errors were ignored or dismissed
  • ‚ùå Poor user interface - cryptic error codes like "Malfunction 54"
  • ‚ùå Insufficient testing for race conditions
üíî Result: Multiple patient deaths and severe injuries
Source: "Medical Devices: The Therac-25" - Nancy Leveson, University of Washington

The Therac-25 Lessons ‚öñÔ∏è

What Software Engineering Teaches Us:

Error Handling Matters
The software dismissed critical errors with codes users didn't understand. Always handle errors gracefully and clearly.

Test Edge Cases
Race conditions only happened with specific timing. Test unusual scenarios, not just happy paths.

Never Trust Software Alone
Critical systems need multiple layers of protection. Hardware interlocks + software validation.

User Communication
Cryptic error messages like "Malfunction 54" prevented proper response. Error messages should be clear and actionable.

Handling Errors Gracefully üõ°Ô∏è

  // Code that might throw an error
