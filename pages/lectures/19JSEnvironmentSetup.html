What is Node.js? 🤔

JavaScript Outside the Browser!

Browser JavaScript

  • Runs in Chrome, Firefox, etc.
  • Access to DOM
  • window object
  • Client-side only

Node.js JavaScript

  • Runs on your computer
  • Access to file system
  • process object
  • Server-side capable
Think of it this way: Node.js lets JavaScript escape the browser sandbox!

Why Do We Need Node.js? 💡

Modern Web Development Requirements

  • 📦 Package Management: Install and manage libraries (React, Vue, etc.)
  • 🛠️ Build Tools: Compile, bundle, and optimize your code
  • 🧪 Testing: Run automated tests for your JavaScript
  • 🖥️ Server Development: Build APIs and backend services
  • ⚡ Development Tools: Live reload, linting, formatting

🕰️ Historical Perspective

FORTRAN Introduced Variables in 1957

1957 → IBM → FORTRAN

The Birth of Modern Programming

  • FORTRAN (FORmula TRANslation) was developed by IBM in 1957
  • First high-level programming language to introduce the concept of variables as we know them today
  • Before FORTRAN, programmers had to work directly with memory addresses
  • Revolutionary: Programmers could use symbolic names like temperature instead of memory locations like 0x7FFF
Source: Computer History Museum

🕰️ Historical Perspective (continued)

From Memory Addresses to Modern Development

Before FORTRAN (1950s):

LOAD 0x2000
ADD 0x2004
STORE 0x2008

⬇️

With FORTRAN (1957+):

sum = a + b

Connection to Today's Lesson

Just as FORTRAN abstracted away memory addresses, modern development environments like Node.js abstract away complexity, letting us focus on solving problems rather than managing low-level details.

Understanding the Tools 🔧

NVM vs Node vs NPM - What's What?

NVM 🔄

Node Version Manager

Manages multiple Node.js versions

Like a wardrobe for Node versions

Node.js ⚙️

JavaScript Runtime

Executes JavaScript code

The actual engine

NPM 📦

Node Package Manager

Installs JavaScript libraries

Your library shopping cart

Don't Confuse: NVM manages Node versions. NPM manages packages. They sound similar but do different things!

Installation Strategy 📋

Why Use NVM Instead of Direct Installation?

✅ With NVM

  • Switch Node versions easily
  • Test on multiple versions
  • No permission issues
  • Project-specific versions
  • Easy upgrades

❌ Without NVM

  • Stuck with one version
  • Complex upgrade process
  • Potential permission problems
  • System-wide installation only
  • Harder to troubleshoot

Pre-Installation Check 🔍

Do You Already Have Node?

Let's check if you already have Node.js installed:

-bash: node: command not found

$ which node

If you see a version number: You have Node! But we'll still install NVM for better management.

Installing NVM on Linux 🐧

Step 1: Download and Install Script

curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.39.0/install.sh | bash

Step 2: Add to Shell Profile

export NVM_DIR="$HOME/.nvm"
[ -s "$NVM_DIR/nvm.sh" ] && \. "$NVM_DIR/nvm.sh"
[ -s "$NVM_DIR/bash_completion" ] && \. "$NVM_DIR/bash_completion"

Step 3: Reload Configuration

source ~/.bashrc
Ubuntu/Debian Users: You might need to install curl first: sudo apt install curl

Installing NVM on macOS 🍎

Step 1: Install Homebrew (if needed)

/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

Step 2: Install NVM via Homebrew

brew install nvm

Step 3: Create NVM Directory & Configure

mkdir ~/.nvm
echo 'export NVM_DIR="$HOME/.nvm"' >> ~/.zshrc
echo '[ -s "/opt/homebrew/opt/nvm/nvm.sh" ] && \. "/opt/homebrew/opt/nvm/nvm.sh"' >> ~/.zshrc
source ~/.zshrc
Note: macOS uses zsh by default. If using bash, replace .zshrc with .bash_profile

Verifying NVM Installation ✅

Make Sure Everything Works!

$ nvm --version
0.39.0

$ command -v nvm
nvm

$ nvm list
No versions installed yet

Troubleshooting:

  • If "command not found": Close and reopen terminal
  • Still not working? Run: source ~/.bashrc or source ~/.zshrc

Installing Node.js with NVM 📥

Step 1: Install LTS Version

$ nvm install --lts
Downloading and installing node v20.11.0...
##################################################### 100.0%
Computing checksum with sha256sum
Checksums matched!
Now using node v20.11.0 (npm v10.2.4)

What is LTS?

Long Term Support - Stable version supported for 30 months. Best for production!

Setting Your Node Version ⚙️

Step 2: Configure Default Version

$ nvm use --lts
Now using node v20.11.0 (npm v10.2.4)

$ nvm alias default lts/*
default -> lts/* (-> v20.11.0)

$ node --version
v20.11.0

$ npm --version
10.2.4
Success! You now have Node.js and npm installed! 🎉

🕰️ Historical Perspective

The Birth of the += Operator

1969 → Bell Labs → B Language

A Revolution in Code Efficiency

  • Compound assignment operators (+=, -=, *=, /=) were first introduced in the B programming language
  • Created by Ken Thompson at Bell Labs around 1969
  • These operators were revolutionary for two key reasons:
    • Readability: x += 5 is clearer than x = x + 5
    • Efficiency: The compiler could optimize these operations better
  • Later inherited by C (1972), then C++, Java, JavaScript, and most modern languages
Source: The Development of the C Language - Dennis M. Ritchie

🕰️ Historical Perspective (continued)

Why JavaScript Includes These Shortcuts

Without Compound Operators:

counter = counter + 1;
total = total + price;
score = score * multiplier;

⬇️

With Compound Operators:

counter += 1; // or counter++
total += price;
score *= multiplier;

Connection to Today's Environment

JavaScript inherited these elegant shortcuts from its programming language ancestors. When you use += in Node.js, you're using syntax that's over 50 years old - a testament to good design!

🕰️ Historical Perspective

The == vs === Debate

1995 → JavaScript → Two Equality Operators

A Source of Confusion Since Day One

  • JavaScript has TWO equality operators: == and ===
  • == performs type coercion: "5" == 5 returns true
  • === checks both value and type: "5" === 5 returns false
  • This dual system has been a source of bugs and confusion for 30 years

[] == false // true (what?!)
"0" == 0 // true (huh?)
null == undefined // true (wait...)
[] === false // false (better)
"0" === 0 // false (expected)
null === undefined // false (clear!)

Source: JavaScript: The Good Parts - Douglas Crockford

🕰️ Historical Perspective (continued)

Why Your Environment Should Catch These

Douglas Crockford's Recommendation

In his influential book "JavaScript: The Good Parts" (2008), Douglas Crockford famously recommended:

"Always use === to avoid type coercion surprises"

This became the standard advice for all JavaScript developers and is now enforced by modern linters.

Connection to Today's Lesson

This is exactly why configuring your environment with tools like ESLint is so important! A properly configured linter will:

  • Warn you when you use == instead of ===
  • Catch other common JavaScript pitfalls
  • Save you hours of debugging weird type coercion bugs

Troubleshooting Common Issues 🔧

Issue 1: "nvm: command not found"

Solution: Source your profile file or restart terminal

source ~/.bashrc or source ~/.zshrc

Issue 2: "Permission denied" errors with npm

Solution: Never use sudo with npm! NVM prevents this issue.

Issue 3: Node version keeps resetting

Solution: Set a default: nvm alias default node

Golden Rule: When in doubt, close terminal and reopen!

Development vs Deployment 🚀

Important Considerations

Development Environment

  • Use NVM for flexibility
  • Install dev dependencies freely
  • Hot reload, debugging tools
  • Source maps enabled
  • Your local machine

Production Environment

  • Fixed Node version
  • Only production dependencies
  • Optimized and minified code
  • Error logging configured
  • Server or cloud platform
⚠️ Remember: "It works on my machine!" → "It works in production!" requires careful planning!

Best Practices ✅

Setting Yourself Up for Success

  • 📌 Use NVM: Always manage Node versions with NVM
  • 🎯 LTS Versions: Stick to LTS for important projects
  • 📦 package-lock.json: Always commit this file to git
  • 🚫 node_modules: Never commit this folder (add to .gitignore)
  • 📝 .nvmrc: Use this file to specify Node version per project
  • 🔐 Environment Variables: Learn about .env files early
  • 🧪 Test Locally: Always test before deploying
