Boolean Values ‚úÖ‚ùå

True or False - That's It!

// Boolean values
let isStudent = true;
let isGraduated = false;

// Comparison operators return booleans
let age = 20;
console.log(age > 18);      // true
console.log(age === 20);    // true (strict equality)
console.log(age !== 25);    // true (not equal)

// Common comparisons
console.log(10 > 5);        // true
console.log(10 < 5);        // false
console.log(10 >= 10);      // true
console.log(10 <= 9);       // false
console.log(10 === "10");   // false (different types!)
console.log(10 == "10");    // true (type coercion - avoid!)
Always use === and !== for comparison! Avoid == and != which do type coercion.

ü¶ã Historical Moment: The First Actual Computer Bug

ü¶ó
September 9, 1947: A team of computer scientists and engineers at Harvard University reported finding a moth trapped between relay contacts of the Harvard Mark II computer in Cambridge, Massachusetts.
They carefully removed the moth and taped it into the logbook with the now-famous note: "first actual case of bug being found."
But wait! The term "bug" in engineering actually dates back much further. Thomas Edison wrote about "'Bugs' as such little faults and difficulties are called" in a letter from 1878!

Multiple Cases, One Variable

Many If/Else

} else if (day === "Friday") {

Switch Statement

let day = "Monday";

switch(day) {
    case "Monday":
        console.log("Start of work week");
        break;
    case "Friday":
        console.log("TGIF!");
        break;
    case "Saturday":
    case "Sunday":
        console.log("Weekend!");
        break;
    default:
        console.log("Regular day");
}

Switch Statement Deep Dive üîç

let grade = "B";

switch(grade) {
    case "A":
        console.log("Excellent!");
        console.log("Keep up the great work!");
        break;  // Exit the switch!
    
    case "B":
    case "C":
        console.log("Good job!");
        break;  // If you forget break, execution continues!
    
    case "D":
        console.log("You can do better");
        break;
    
    default:  // Like 'else' - runs if no case matches
        console.log("Invalid grade");
}
Critical: Don't forget break! Without it, execution "falls through" to the next case.

Ternary Operator ‚ùì

Shorthand If/Else

Regular If/Else

let age = 20;
let canVote;

if (age >= 18) {
    canVote = "Yes";
} else {
    canVote = "No";
}

console.log(canVote);  // "Yes"

Ternary Operator

let age = 20;

let canVote = age >= 18 ? "Yes" : "No";

console.log(canVote);  // "Yes"

// Syntax: condition ? valueIfTrue : valueIfFalse
Use ternary for simple conditions. For complex logic, stick with if/else for readability!

Comparison Operators Cheat Sheet ‚öñÔ∏è

// Equality (ALWAYS USE STRICT!)
5 === 5        // true  - Strict equality (type + value)
5 === "5"      // false - Different types
5 == "5"       // true  - Loose equality (AVOID!)

5 !== 6        // true  - Strict inequality
5 != "5"       // false - Loose inequality (AVOID!)

// Relational
10 > 5         // true  - Greater than
10 < 5         // false - Less than
10 >= 10       // true  - Greater than or equal
10 <= 9        // false - Less than or equal

// String comparison (alphabetical)
"apple" < "banana"  // true
"Z" < "a"           // true (uppercase comes before lowercase)

Common Conditional Patterns üé®

// Checking for existence
let username = getUserInput();
if (username) {
    console.log(`Hello, ${username}!`);
} else {
    console.log("Please enter a username");
}

// Range checking
let score = 75;
if (score >= 0 && score <= 100) {
    console.log("Valid score!");
}

// Multiple conditions
if (day === "Saturday" || day === "Sunday" || isHoliday) {
    console.log("No work today!");
}

// Guard clauses (exit early)
function processUser(user) {
    if (!user) return "No user provided";
    if (!user.age) return "Age required";
    if (user.age < 18) return "Must be 18+";
    
    // Main logic here
    return "User processed!";
}

üóìÔ∏è Historical Moment: The Y2K Millennium Bug

The Problem: In the early days of computing, memory was extremely expensive. To save space, programmers abbreviated four-digit years to just two digits.
For example, "1998" was stored as just "98". This worked great... until the year 2000 approached.
The Crisis: As December 31, 1999 approached, there was genuine fear that when clocks rolled over to "00", systems worldwide might interpret this as the year 1900 instead of 2000.
The Impact: The Gartner research firm estimated the global cost of fixing Y2K bugs at $300-$600 billion. Countless programmers worked overtime to patch legacy systems.
üìö Source: National Museum of American History - "Y2K"

üîç Why This Matters Today

üîó The Connection to Our Work:

Y2K demonstrates why proper data type handling and edge case testing are absolutely critical in programming.

When you write conditionals that check ages, dates, or numbers, ask yourself: "What happens at the boundaries?" What if someone enters the year 2100? What if age is 0? What if the input is negative?

Test your edge cases! A small oversight in data type assumptions can lead to massive problems down the road.
