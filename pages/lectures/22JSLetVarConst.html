A Brief History üìú

Why did we need let and const?

Pre-ES6 (Before 2015)

  • Only had var for variables
  • Led to confusing scope issues
  • Caused unexpected bugs
  • Made code harder to reason about

ES6+ (2015 and after)

  • Introduced let and const
  • Block-level scoping
  • More predictable behavior
  • Clearer intent in code

Understanding var üìù

The Original Variable Declaration

var greeting = "Hello World!";
var count = 42;
var isActive = true;

Key Characteristics:

  • üìç Function-scoped or globally-scoped
  • ‚ôªÔ∏è Can be re-declared in the same scope
  • ‚úèÔ∏è Can be updated/reassigned
  • ‚¨ÜÔ∏è Hoisted to the top of their scope
  • üîß Initialized with undefined when hoisted

Scope of var - Global üåç

var globalGreeting = "Hey hi";

function showGreeting() {
  console.log(globalGreeting); // "Hey hi"
}

console.log(globalGreeting); // "Hey hi"
Global Scope (Window)
globalGreeting is available everywhere!
‚ö†Ô∏è Warning: Global variables can be accessed and modified from anywhere, which can lead to bugs!

Scope of var - Function üì¶

function newFunction() {
  var hello = "hello";
  console.log(hello); // "hello"
}

newFunction();
console.log(hello); // ‚ùå Error: hello is not defined
Global Scope
Function Scope
hello only exists here!
‚ùå Cannot access hello out here

var Can Be Re-declared üîÑ

Re-declaration ‚úÖ

var greeter = "hey hi";
var greeter = "say Hello";
console.log(greeter);
// "say Hello"

Update ‚úÖ

var greeter = "hey hi";
greeter = "say Hello";
console.log(greeter);
// "say Hello"
Note: Both work with var, but re-declaration can cause confusion!

Hoisting with var ‚¨ÜÔ∏è

Variables are moved to the top!

What you write:

console.log(greeter);
var greeter = "say hello";

How JS interprets it:

var greeter; // Declaration hoisted
console.log(greeter); // undefined
greeter = "say hello"; // Assignment stays
Result: No error, but logs undefined instead of the value!

The Problem with var ‚ö†Ô∏è

var greeter = "hey hi";
var times = 4;

if (times > 3) {
  var greeter = "say Hello instead";
}

console.log(greeter); // "say Hello instead" üò±

What happened?

var doesn't have block scope! The greeter inside the if-block overwrote the outer one!

This is why we needed let and const!

Introducing let üéâ

The Modern Variable Declaration

let message = "Hello";
let count = 100;
let isReady = false;

Key Improvements over var:

  • üéØ Block-scoped (respects { } boundaries)
  • ‚ùå Cannot be re-declared in same scope
  • ‚úÖ Can be updated/reassigned
  • ‚ö†Ô∏è Hoisted but not initialized (Temporal Dead Zone)
Best Practice: Use let when the value will change later!

let is Block Scoped üéØ

let greeting = "say Hi";
let times = 4;

if (times > 3) {
  let hello = "say Hello instead";
  console.log(hello); // "say Hello instead"
}

console.log(hello); // ‚ùå Error: hello is not defined
Outer Scope
Block Scope (if statement)
hello only exists in here!
‚ùå hello doesn't exist out here!

let Cannot Be Re-declared üö´

Re-declaration ‚ùå

let greeting = "say Hi";
let greeting = "say Hello";
// ‚ùå Error: Identifier 'greeting'
// has already been declared

Update ‚úÖ

let greeting = "say Hi";
greeting = "say Hello";
console.log(greeting);
// ‚úÖ "say Hello"
Benefit: Prevents accidental variable name collisions!

let in Different Scopes üì¶

let greeting = "say Hi";

if (true) {
  let greeting = "say Hello instead";
  console.log(greeting); // "say Hello instead"
}

console.log(greeting); // "say Hi" ‚úÖ

Why no error?

These are two different variables in different scopes! The inner greeting shadows the outer one only within the block.

Temporal Dead Zone ‚ö†Ô∏è

The danger zone for let and const!

What is TDZ?

The time between entering scope and the variable declaration where the variable cannot be accessed.

console.log(myVar); // ‚úÖ undefined (var)
console.log(myLet); // ‚ùå ReferenceError (let)

var myVar = 5;
let myLet = 10;
Why TDZ exists: Makes errors easier to catch - accessing variables before declaration is usually a mistake!

Introducing const üîí

For Values That Don't Change

const PI = 3.14159;
const API_KEY = "abc123xyz";
const MAX_SIZE = 100;

Key Characteristics:

  • üéØ Block-scoped (like let)
  • ‚ùå Cannot be re-declared
  • ‚ùå Cannot be reassigned
  • üìù Must be initialized at declaration
  • ‚ö†Ô∏è Also has Temporal Dead Zone
Best Practice: Use const by default, switch to let only when needed!

const Rules üìã

‚ùå What You Can't Do

const greeting = "Hi";
greeting = "Hello"; // Error!

const greeting = "Hi";
const greeting = "Hey"; // Error!

const name; // Error!
name = "John";

‚úÖ What You Must Do

// Initialize at declaration
const greeting = "Hi";

// Use in different scopes
if (true) {
  const greeting = "Hello";
}

const with Objects & Arrays üé®

The reference is constant, not the contents!

Objects

const person = {
  name: "Alice",
  age: 30
};

// ‚úÖ Can modify properties
person.age = 31;
person.city = "NYC";

// ‚ùå Can't reassign
person = {}; // Error!

Arrays

const numbers = [1, 2, 3];

// ‚úÖ Can modify contents
numbers.push(4);
numbers[0] = 10;

// ‚ùå Can't reassign
numbers = [5, 6]; // Error!
Remember: const prevents reassignment of the variable itself, not modification of object/array contents!

Quick Comparison ‚öñÔ∏è

When to use each declaration

var

‚ùå Almost never! Only in legacy code or very specific cases

let

‚úÖ When you need to reassign the variable

  • Loop counters: for (let i = 0; ...)
  • Accumulator values
  • Conditional assignments

const

‚úÖ Default choice! Use unless you need to reassign

  • Configuration values
  • DOM element references
  • Imported modules
  • Most variables in modern JS

Practical Examples üíª

Real-world usage patterns

// Configuration values - use const
const API_URL = "https://api.example.com";
const MAX_RETRIES = 3;

// DOM elements - use const
const button = document.querySelector('.btn');
const container = document.getElementById('app');

// Values that change - use let
let currentUser = null;
let isLoading = false;

// Loops - use let for counters
for (let i = 0; i < 10; i++) {
  console.log(i);
}

Common Pitfalls üï≥Ô∏è

Mistakes to avoid

1. Using var in modern code

// ‚ùå Don't do this
var name = "John";

// ‚úÖ Do this instead
const name = "John";

2. Using let when const would work

// ‚ùå Unnecessary let
let PI = 3.14159;

// ‚úÖ Use const for constants
const PI = 3.14159;

3. Accessing before declaration

// ‚ùå TDZ error
console.log(value);
const value = 42;

Best Practices ‚ú®

Modern JavaScript Guidelines

The Modern Approach:

  1. üîí Use const by default
  2. üîÑ Use let when you need to reassign
  3. ‚ùå Avoid var completely
  4. üìù Declare variables at the top of their scope
  5. üéØ Keep variables in the smallest scope possible
  6. ‚úÖ Initialize variables when you declare them
Remember: Using const doesn't mean the value is immutable, it means the binding is immutable!

Let's Practice! üèãÔ∏è

Refactoring Exercise

Refactor this code to use let and const:

var userAge = 25;
var userName = "Alice";
var scores = [85, 92, 78];
var total = 0;

for (var i = 0; i < scores.length; i++) {
  total = total + scores[i];
}

var average = total / scores.length;
userAge = userAge + 1;

Take 5 minutes to refactor this code!

Exercise Solution ‚úÖ

let userAge = 25; // Will be reassigned
const userName = "Alice"; // Won't change
const scores = [85, 92, 78]; // Array ref won't change
let total = 0; // Will accumulate

for (let i = 0; i < scores.length; i++) { // Loop counter
  total = total + scores[i];
}

const average = total / scores.length; // Calculated once
userAge = userAge + 1; // Reassignment works with let
Analysis: We used let for values that change (userAge, total, i) and const for everything else!

Summary üìö

Key Takeaways

Remember:

  • var: Function-scoped, avoid in modern code
  • let: Block-scoped, use for values that change
  • const: Block-scoped, use by default
  • Hoisting: All are hoisted, but let/const have TDZ
  • Best Practice: const ‚Üí let ‚Üí never var
Pro Tip: When in doubt, start with const. Your IDE will tell you if you need let!
